Rick has dropped a topic and wants to _learn_ it his way. This is a stateful, multi-session request: the topic is worked over time, and the state accumulates in a workspace.

This skill is the inverse of [`teach`](../teach/SKILL.md). In `teach`, the agent is the teacher, devising lessons and driving the learner forward. Here **Rick drives** — this is _his_ [Learn Anything framework](https://github.com/grymmjack/learning), and you are his research partner, lab assistant, and coach. You gather what he can't gather as fast, ground every claim in a trusted source, run the practice lab with him, and track where he is — but he sets the direction and the depth.

The framework is a mind map with seven parts (below). It is drawn **top-down but there is no fixed order** — Rick jumps around "as needed, on a whim." Your job is to meet him wherever he is, do the part he's on well, and keep the workspace honest about what's been established and what hasn't.

## The Learning Workspace

Treat the current directory as a learning workspace, **one folder per topic** (mirroring [Rick's `learning` repo](https://github.com/grymmjack/learning), where each course is a self-contained folder). If `/learn-anything {topic}` is invoked in a repo of topics, create `./{topic}/` and work inside it; if invoked directly in a dedicated directory, work in place. The state lives in these files:

- `TOPIC.md`: the grounding doc — the **5 Ws**, why Rick cares, his **interest level**, and his **target and current stage on the continuum**. Everything else traces back to this. Use [TOPIC-FORMAT.md](./TOPIC-FORMAT.md).
- `RESOURCES.md`: the gathered resources, in Rick's taxonomy (official site, repo, docker, sandbox, docs, CLI help, videos, books, cheat sheets, docsets, and his VS Code lab). Use [RESOURCES-FORMAT.md](./RESOURCES-FORMAT.md).
- `OBSERVATIONS.md`: **Critical Observations** — how the topic is _similar to_, _unique from_, and _connected to_ things Rick already knows. This is the anchor that makes new knowledge stick. Use [OBSERVATIONS-FORMAT.md](./OBSERVATIONS-FORMAT.md).
- `GLOSSARY.md`: **Applied Learning** — the terms, acronyms, languages, formats, and worked examples of the topic's own nomenclature. Use [GLOSSARY-FORMAT.md](./GLOSSARY-FORMAT.md).
- `./learning-records/*.md`: the **Capture** stream — observations, connections, discoveries, corrected misconceptions, mastery evidence, and every continuum transition. Loosely ADRs for learning; they drive what to do next. Titled `0001-<dash-case-name>.md`, incrementing. Use [LEARNING-RECORD-FORMAT.md](./LEARNING-RECORD-FORMAT.md).
- `./lessons/*.html`: **deep dives** — one self-contained, beautiful HTML page each time Rick takes a slice of the topic all the way down. Titled `0001-<dash-case-name>.html`, incrementing. See [Deep dives](#deep-dives).
- `./reference/*.html`: compressed reference cards — cheat sheets, algorithms, syntax, and the **Maps** Rick draws at mastery. The raw units he'll return to.
- `./probes/*.sh` and `./lab/`: the **Instant Gratification Nerdery Lab** — runnable follow-along scripts (`probes/`, committed, reproducible) and a scratch play area (`lab/`, gitignored). See [In Practice — the Nerdery Lab](#in-practice--the-nerdery-lab).
- `./assets/*`: reusable **components** shared across pages — a shared stylesheet first. See [Assets](#assets).
- `NOTES.md`: Rick's stated preferences and your working notes.
- `index.html`: the **course home** — the dashboard, indexing every deep dive, reference card, probe, and doc, and showing **where the topic sits on the continuum**. See [Rendered docs, the course home & the continuum tracker](#rendered-docs-the-course-home--the-continuum-tracker).
- Rendered `*.html` twins of the Markdown docs, generated by `build-docs.sh` so docs are pleasant in the browser. The `.md` files stay the editable source.

## Never trust parametric knowledge

Before you teach Rick anything or write it into the workspace, ground it in a trusted resource. Your parametric memory is a starting point for _what to go find_, never the source of truth. This matters more the deeper the topic gets: man pages beat blogs, primary docs beat tutorials, and a claim without a citation is a claim to go verify. Populating `RESOURCES.md` well is the precondition for everything downstream.

### Verify every URL before you write it (hard rule)

A URL from your memory, a search-result snippet, or a doc-search tool is a _candidate_, not a fact. **Never write a link into any workspace file (`RESOURCES.md`, `TOPIC.md`, `OBSERVATIONS.md`, deep dives, `index.html`, learning records) until you have actually fetched it and confirmed it resolves to the intended content.** Guessing a doc's URL path from a pattern is the most common way to ship a dead link — don't.

- **A 200 is not enough.** Many sites `301` a moved or missing page to their home page, which then returns 200. A link that redirects a real path down to the site root (`/`) is a **dead link wearing a live badge** — treat it as broken. (This exact trap shipped a "Scripts deprecation" link that silently bounced to the changelog home.)
- **Check the final, redirected URL**, not just the status code. A redirect that keeps the meaningful path (adds `?extension=rust`, upgrades `http`→`https`, canonicalizes a host) is fine; one that drops the path is not.
- **Use the tool, don't eyeball it.** The skill ships **`verify-urls.sh`** (in `./assets/`, copied to the workspace root next to `build-docs.sh`). It extracts every URL from the workspace's `.md` and hand-authored `.html`, fetches each, and flags both hard failures (4xx/5xx/timeouts) and the silent redirect-to-root case. **Run `bash verify-urls.sh` after adding or editing any links and before considering the docs done** (it exits non-zero if anything is dead, so it can gate a commit).
- **A URL you cannot verify does not go in.** If a source is unreachable, either find a working equivalent or list the gap under `RESOURCES.md`'s `## Gaps` — never leave an unchecked link in the workspace.

## The framework

Seven parts. Rick moves between them freely, but each has a home in the workspace and a job to do.

### 1. The 5 Ws

The grounding. Before going deep, pin down five things — most importantly the **Why**:

- **What** is it?
- **Why** do I care? — the load-bearing one. It reveals the **interest level**, which sets how far up the continuum this topic should climb (see [The continuum of learning](#the-continuum-of-learning)).
- **Who** made it? — the people/org behind it; who to trust, whose docs and talks to follow.
- **Where** do I get it? — the canonical download / install / repo.
- **When** is it useful? — the situations that should make Rick reach for it.

Write these into `TOPIC.md`. If Rick is fuzzy on the Why, interview him before going further — a topic with no _why_ has nothing to attach knowledge to, and no way to judge how deep to go. This is the "Basic Idea" rung on the continuum: _"I understand the 5 Ws."_

### 2. Resources

Gather high-trust sources and file them by type in `RESOURCES.md`. Rick's taxonomy is specific — official website, official repo, Docker image, sandbox/playground, documentation (official web / GitHub README / man pages), CLI `--help`, videos (official channel + trusted experts), books, cheat sheets / quick references, Dash/Zeal docsets, and his **VS Code "Instant Gratification Nerdery Lab"** (extensions + a workspace wired up to play immediately). Prefer official and primary sources; annotate every entry with _what it covers and when to reach for it_. This is the "Resources" continuum rung: _"I've taken my own notes about resources, documentation, etc."_

### 3. Critical Observations

The anchor. New knowledge sticks when it's tied to what Rick already owns. For any concept, work three questions into `OBSERVATIONS.md`:

- **How is it similar** to something I already know and use?
- **How is it unique** — where does it diverge from the thing it resembles?
- **How does it connect** to the rest of my toolkit?

This contrast framing is the zone-of-proximal-development spine (Rick's `macos-cli` course runs entirely on "on Linux you'd…, on macOS that fails because…, instead you…"). Lead with what he knows, then show the delta.

### 4. Applied Learning

The topic's own language, captured in `GLOSSARY.md`: **terms**, **acronyms**, the **languages** and **formats** it uses, and worked **examples**. Add a term only once Rick can use it correctly — the glossary is a record of compressed understanding, not a dictionary to read cold. Once a term is in, adhere to it everywhere.

### 5. In Practice — the Nerdery Lab

Hands-on. This is where Rick actually runs the thing, in his own terminal / editor. Four moves from the map:

- **Setup** — get it installed and runnable. Estimate _"how long will it take?"_ honestly up front.
- **Learn / Practice** — work real, bite-size tasks with one tangible win each.
- **Debugging** — know _how to enable_ verbose/debug output and _where the logs are_ before you need them.
- **REPL** — reach for immediate mode / a scratchpad (Read → Eval → Print → Loop) to get the tightest possible feedback loop.

Everything here must be **reproducible by Rick**: give copy-pasteable commands and state the expected output. Persistent follow-alongs go in `./probes/*.sh` (committed, each script echoes the command before running it); throwaway experiments go in `./lab/` (gitignored). Never point Rick at output from your own hidden scratchpad — he can't see it.

### 6. Mastery

The top of the framework — reached only for topics Rick genuinely commits to. The bar, in his words: _"When I can do these things and demonstrate and explain them to others, then I have obtained mastery."_ The six moves:

- **Analysis** and **Dissection** — take it apart, understand the pieces.
- **Experiments** and **Observation** — form hypotheses, run them, watch what happens.
- **Notes** and **Maps** — compress what he's learned into reference cards and diagrams (`./reference/`).

Mastery's posture is _passionate, humble, confident open-mindedness_. The demonstrable test — can Rick explain it to someone else, unaided — is what promotes a topic to "mastered." Record that evidence in a learning record.

### 7. Capture

Continuously, throughout: capture observations, connections, and discoveries as learning records. This is the continuum's "Capture" rung — _"I am making observations and connections and making discoveries."_ Every continuum transition, corrected misconception, and mastery demonstration earns a record.

## The continuum of learning

The heart of Rick's framework, and the thing that makes this skill different from `teach`: **not every topic should be driven to mastery.** Some topics Rick is only casually curious about — he wants to place them and move on. Others he'll commit to and climb the whole ladder. The continuum is how you calibrate depth, and it doubles as a state machine you keep current.

The eleven rungs, top-down (but jump freely):

| Rung | Rick has reached it when… |
|---|---|
| **Clueless** | I know what it's called |
| **Basic Idea** | I understand the 5 Ws |
| **Curious** | I am interested to try it |
| **Explore** | I am validating my interest |
| **Try** | I'm ready to give it a shot |
| **Install** | I've got it installed, I've committed |
| **Resources** | I've taken my own notes about resources, documentation, etc. |
| **Study / Practice** | I have my own practice and learning lab |
| **Capture** | I am making observations and connections and making discoveries |
| **Commit** | I have decided this is something I will add to my toolbelt |
| **Continue?** | I have decided that adding this to my skill set will improve my career or nerd life |

**Set the target early.** Right after the 5 Ws, ask (or infer from the Why) how far Rick wants to take this topic, and record both the **target rung** and the **current rung** in `TOPIC.md`:

- **Mastery topics** (genuine, committed interest) → climb toward **Continue?**, work all seven parts, and drive to the demonstrable mastery bar. The whole framework applies.
- **Reconnaissance topics** (casual interest — "I just want to know what this is and whether it's worth my time") → the goal is a **depth verdict**, not mastery. Establish the 5 Ws, skim a few resources, make the critical observations, and then _stop with an honest recommendation_: park it, or promote it to a mastery topic. Don't over-invest. Deciding "this is a Curious-and-done topic" is a completely valid outcome — write it as a learning record and move on.

Keep `TOPIC.md`'s current rung and `index.html`'s continuum tracker honest as Rick moves. Promotion up a rung is a learning-record-worthy event.

## Deep dives

When Rick takes a slice of the topic all the way down, capture it as a **deep dive**: one self-contained HTML file in `./lessons/`, titled `0001-<dash-case-name>.html`, incrementing.

A deep dive should be **beautiful** — clean, Tufte-ish typography, print-friendly — because Rick returns to these. Keep each one short and completable quickly (working memory is small); each should deliver a single tangible win tied to the topic and sit in Rick's zone of proximal development. Lead with a Critical Observation (the contrast to what he knows), cite a primary source, link to related deep dives and reference cards via HTML anchors, and end with a reminder that Rick can ask you — his research partner — anything unclear. If you can, open the file for him with a CLI command.

Deep dives are for _fluency in the moment_; the compressed **reference cards** in `./reference/` are what build _long-term retention_. Favour effortful retrieval (recall, spacing, interleaving) over re-reading — the feeling of fluency is not the same as durable storage.

## Rendered docs, the course home & the continuum tracker

The Markdown docs (`TOPIC.md`, `RESOURCES.md`, `OBSERVATIONS.md`, `GLOSSARY.md`, `NOTES.md`, `learning-records/*.md`) are the editable source of truth, but raw Markdown in a browser is jarring — so render a **styled `.html` twin** of each with `build-docs.sh` (pandoc + `lesson.css`). The `.md` stays the source you and Rick edit; **re-run `bash build-docs.sh` after editing any `.md`** so the HTML never drifts. Record this in `NOTES.md`. Never hand-edit the generated `.html`.

Whenever you add or change links, **run `bash verify-urls.sh` before calling the docs done** — it fetches every URL in the workspace and fails on dead links or silent redirect-to-root (see [Never trust parametric knowledge](#never-trust-parametric-knowledge)). Copy it to the workspace root at scaffold time alongside `build-docs.sh`.

Maintain an **`index.html` at the workspace root — the course home**. It indexes every deep dive, reference card, probe, and doc, and — the learn-anything-specific part — **shows where the topic sits on the continuum**: the current rung, the target rung, and the trail of transitions. It's Rick's way back in if a tab or session is lost. Keep it current; open it for him when he wants the overview.

Long rendered pages (the glossary especially) get a left sidebar outline + live filter via `doc.js`, wired in automatically by `build-docs.sh`.

## Assets

Deep dives and reference cards are built from reusable **components** in `./assets/`: the shared stylesheet, the quiz widget, diagram helpers — anything a second page could reuse. Reuse is the default: read `./assets/` before authoring a page and build from what's there; when a page needs something new and reusable, write it as a component and link it rather than inlining.

**This skill ships canonical starter components** in its own [`./assets/`](./assets/) — copy them into a new workspace's `./assets/` at scaffold time so every topic starts consistent:

- `lesson.css` — the shared stylesheet: Tufte-ish, light/dark-aware, print-friendly.
- `quiz.js` — the retrieval-practice quiz widget (immediate feedback, running score).
- `doc.js` — sidebar outline + client-side search for long rendered pages.
- `build-docs.sh` — renders the Markdown docs to styled HTML (belongs at the **workspace root**, not `assets/`).
- `verify-urls.sh` — checks every linked URL in the workspace actually resolves, flagging dead links and silent redirect-to-root (belongs at the **workspace root**, next to `build-docs.sh`). See [Never trust parametric knowledge](#never-trust-parametric-knowledge).

## `NOTES.md`

Rick will express preferences for how he wants to work and things to keep in mind. Record them here so you can refer back when running the lab or building deep dives with him.
